## Context

当前系统基于关键词监控商品，缺少以下能力：
- 无法按商家维度批量监控店铺商品
- 无法通过商品 ID 精确查找
- Web UI 无法像桌面应用一样独立窗口运行
- 没有追踪商品价格/想要数变化并推送通知

技术栈：FastAPI 后端 + Vue 3 前端 + Playwright 爬虫 + 多模态 AI 分析

## Goals / Non-Goals

**Goals:**
- 实现商家维度监控：输入卖家 ID，自动抓取并监控其店铺所有商品
- 实现商品 ID 精确搜索：通过闲鱼商品 ID 直接定位
- PWA 化：支持安装为桌面应用，独立窗口打开
- 实现价格/想要数变化检测与通知
- 集成飞书/钉钉 Webhook 通知渠道

**Non-Goals:**
- 不重构现有关键词监控逻辑
- 不改变现有数据存储结构（仅新增表）
- 不支持其他平台（如转转、淘宝）的商家监控

## Decisions

### 1. 商家监控实现方式
**决策**: 通过闲鱼搜索结果中的卖家 ID 过滤实现，而非爬取整个店铺
- **方案 A**: 搜索关键词 + 卖家 ID 过滤（采用）
- **方案 B**: 直接爬取卖家店铺主页
- **选择理由**: 方案 A 更稳定，闲鱼店铺页 anti-scraping 较强；方案 B 需要额外处理店铺页翻页和反爬

### 2. PWA 实现方案
**决策**: 使用 vite-plugin-pwa 插件
- **方案 A**: vite-plugin-pwa（采用）
- **方案 B**: 手动编写 manifest + Service Worker
- **选择理由**: 项目已使用 Vite，插件集成更简单，支持自动更新

### 3. 指标变化检测策略
**决策**: 每次爬取后对比历史快照，变化超过阈值触发通知
- 价格变化：任意变化即触发（或设置百分比阈值）
- 想要数变化：设置增量阈值（如 +10、+50）
- 历史记录保存最近 30 天，每分钟粒度

### 4. 通知触发架构
**决策**: 在 TaskService 中嵌入指标检测逻辑，异步推送
- **方案 A**: 爬取完成后同步检测（采用）
- **方案 B**: 独立 MetricsCheckService 定时扫描
- **选择理由**: 方案 A 实时性更好，避免额外定时任务复杂度

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 卖家 ID 搜索可能受闲鱼反爬限制 | 复用现有 Playwright 浏览器上下文，保持登录状态 |
| PWA 在部分桌面浏览器兼容性差 | 降级为普通网页，核心功能不受影响 |
| 指标历史数据量增长快 | 定期清理 30 天前数据，支持配置保留周期 |
| 飞书/钉钉 Webhook 限流 | 实现重试机制和队列缓冲 |
| 商家商品数量过多导致爬取超时 | 限制单次最多抓取 50 件商品，分页需手动触发 |

## Migration Plan

1. 数据库迁移：执行 SQL 脚本创建 seller_info、item_metrics_history 表
2. 后端更新：部署新 Service 和 API 路由
3. 前端更新：构建 PWA 并推送静态资源
4. 配置更新：在 `.env` 中添加飞书/钉钉 Webhook URL
5. 回滚：还原代码至上一版本，删除新增表（可选保留数据）
